## 2. 企画概要

本サービスは、近所のスーパーやドラッグストアなどの価格・特売情報をもとに、ユーザーの買い物判断を支援するWebサービスである。

ユーザーは買いたい商品を買い物リストとして登録し、システムは登録済みの商品価格データをもとに、合計金額、店舗数、移動時間を考慮した買い物プランを提案する。

本サービスは、単なるチラシ閲覧サービスや価格比較サービスではなく、ユーザーが「どこで、どのように買えばよいか」を判断するための買い物判断支援サービスを目指す。

---

## 3. 背景

近年、物価上昇により、日常の買い物において少しでも安く商品を購入したいというニーズが高まっている。

一方で、スーパーやドラッグストアごとの価格・特売情報は、チラシ、各店舗のWebサイト、既存の特売情報サービスなどに分散している。

そのため、ユーザーが複数店舗の価格を比較し、どこで購入すれば最も安く、かつ効率的に買い物できるかを判断するには時間と手間がかかる。

また、既存サービスではチラシや特売情報を閲覧できるものの、ユーザーが実際に購入したい商品リストをもとに、合計金額、店舗数、移動時間を考慮した買い物方法を提示する点には改善の余地がある。

## 4. 解決したい課題

本サービスで解決したい課題は、ユーザー側の課題と店舗側の課題に分けられる。

### 4.1 ユーザー側の課題

| 課題                     | 内容                                                       |
| ------------------------ | ---------------------------------------------------------- |
| 価格比較に時間がかかる   | 複数店舗のチラシやWebサイトを個別に確認する必要がある      |
| 買い物判断が難しい       | どの商品をどの店舗で買えばよいか判断しにくい               |
| 合計金額が分かりにくい   | 商品単位では安く見えても、買い物全体で安いか判断しにくい   |
| 移動コストを考慮しにくい | 安さだけでなく、店舗までの距離や移動時間も考える必要がある |
| チラシ確認が面倒         | チラシを一つずつ見て、必要な商品を探す手間がある           |

### 4.2 店舗側の課題

| 課題                   | 内容                                                         |
| ---------------------- | ------------------------------------------------------------ |
| 情報登録に手間がかかる | 商品価格や特売情報を登録・更新する作業が負担になる           |
| 情報発信が継続しにくい | 毎日・毎週の価格変更や特売情報更新に時間がかかる             |
| ユーザーに届きにくい   | 登録した特売情報が必要なユーザーに届いているか分かりにくい   |
| 効果が見えにくい       | 掲載した情報がどの程度見られたか、反応があったか把握しにくい |

ただし、第一版では開発範囲を限定し、まずはユーザー側の買い物判断支援機能を中心に実装する。店舗側の情報登録機能や管理者側の運用機能は、将来的な拡張機能として検討する。

## 5. サービス構築の目的

本サービスの目的は、ユーザーが買いたい商品を登録することで、近所の店舗の価格情報を比較し、最適な買い物方法を分かりやすく提案することである。

単に商品ごとの最安値を表示するだけでなく、買い物リスト全体の合計金額、店舗数、移動時間を考慮し、ユーザーの状況に応じた複数の買い物プランを提示する。

将来的には、店舗側が価格や特売情報を簡単に登録・更新できる仕組みを提供し、ユーザー側の買い物判断と店舗側の情報発信の双方を支援するサービスを目指す。

---

## 6. ゴール

本サービスのゴールは以下の通りである。

- ユーザーが買いたい商品を買い物リストとして登録できる
- 登録した商品について、複数店舗の価格を比較できる
- 商品ごとの最安値を確認できる
- 買い物リスト全体の合計金額を比較できる
- 「時間優先」「最安値優先」「バランス重視」など、複数の買い物プランを提示できる
- 必要に応じて、Googleマップリンクでルートを確認できる
- 就職活動時に、課題発見力、設計力、実装力を示すポートフォリオとして利用できる

## 9. 想定利用シーン

### 9.1 買い物前の価格確認

ユーザーは買い物前に、購入予定の商品を買い物リストに登録する。

システムは登録された商品について、近所のスーパーやドラッグストアの価格情報を比較し、買い物リスト全体の合計金額を表示する。

### 9.2 買い物プランの比較

ユーザーが複数の商品を購入したい場合、システムは以下のような買い物プランを提示する。

| プラン             | 内容                                         |
| ------------------ | -------------------------------------------- |
| 時間優先プラン     | 近い店舗、または一店舗でまとめて買えるプラン |
| 最安値優先プラン   | 商品ごとに最も安い店舗で購入するプラン       |
| バランス重視プラン | 合計金額と移動時間のバランスを考慮したプラン |

これにより、ユーザーは「一店舗でまとめて買う場合」と「複数店舗を回る場合」のどちらが効率的かを判断できる。

### 9.3 ルート確認

複数店舗を回る場合、第一版ではGoogleマップのURLリンクを表示し、ユーザーが外部サービス上でルートを確認できるようにする。

将来的には、Google Maps API等を利用し、サイト内で移動時間や距離を表示できるようにする。

## 10. 機能要件

### 10.1 買い物リスト登録機能

ユーザーは購入したい商品を買い物リストに登録できる。

#### 入力項目

| 項目   | 内容                       |
| ------ | -------------------------- |
| 商品名 | 卵、牛乳、米、キャベツなど |
| 数量   | 1個、1本、1kgなど          |
| メモ   | 必要に応じて補足情報を入力 |

---

### 10.2 商品価格比較機能

登録された商品について、複数店舗の価格を比較する。

#### 比較対象

- 商品名
- 価格
- 店舗名
- 規格
- 特売期間
- 更新日

---

### 10.3 最安値表示機能

商品ごとに最も安い店舗を表示する。

#### 表示例

| 商品名 | 最安値 | 店舗名    | 特売期間               |
| ------ | -----: | --------- | ---------------------- |
| 卵     |  198円 | スーパーA | 2026/07/09〜2026/07/11 |
| 牛乳   |  168円 | スーパーB | 2026/07/09〜2026/07/10 |

---

### 10.4 買い物プラン提案機能

買い物リスト全体をもとに、複数の買い物プランを表示する。

#### 表示内容

| 項目         | 内容                                   |
| ------------ | -------------------------------------- |
| プラン名     | 時間優先、最安値優先、バランス重視など |
| 対象店舗     | 購入対象となる店舗                     |
| 合計金額     | 買い物リスト全体の合計金額             |
| 節約金額     | 基準プランとの差額                     |
| 移動時間     | 店舗まで、または店舗間の移動時間       |
| おすすめ理由 | なぜそのプランを提案するかの説明       |

#### 表示例

| プラン             | 内容                                         |
| ------------------ | -------------------------------------------- |
| 時間優先プラン     | スーパーAでまとめて購入。移動時間が短い      |
| 最安値優先プラン   | 商品ごとに最安店舗を選択。合計金額が最も安い |
| バランス重視プラン | 2店舗を回り、価格と移動時間のバランスを取る  |

---

### 10.5 ルート確認機能

複数店舗を回る場合、Googleマップ等を利用してルートを確認できるようにする。

#### 第一版

第一版では、GoogleマップのURLリンクを生成する形式を想定する。

#### 将来的な拡張

将来的には、Google Maps API等を利用し、サイト内で移動時間や距離を表示できるようにする。

---

### 10.6 簡易データ登録機能

第一版では、実際の店舗連携は行わず、開発者または管理者が仮の商品価格データを手入力で登録する。

将来的には、店舗側が自ら価格や特売情報を登録・更新できる店舗向け管理画面を実装する。

---

## 11. 非機能要件

### 11.1 操作性

- スマートフォンでも見やすい画面構成にする
- 商品検索から結果表示までの操作を簡単にする
- 専門知識がなくても利用できるUIにする
- 買い物前に短時間で判断できる画面設計にする

### 11.2 性能

- 商品検索結果は数秒以内に表示されることを目指す
- 第一版では小規模データを対象とするため、大量データ処理は対象外とする

### 11.3 保守性

- 商品情報、店舗情報、価格情報を分けて管理する
- 将来的にデータ取得方法を変更できる構造にする
- 店舗側機能や管理者側機能を追加しやすい構成にする

### 11.4 セキュリティ

- 第一版ではログイン機能を実装しない
- 将来的にユーザー登録や店舗アカウント機能を追加する場合は、認証・個人情報管理を考慮する

## 15. ラフなシステム構成図

本サイトのラフなシステム構成は以下の通りである。

    ユーザー
      ↓
    フロントエンド
      ↓
    バックエンドAPI
      ↓
    データベース
      ├── 店舗情報
      ├── 商品情報
      ├── 価格情報
      └── 買い物リスト情報
      ↓
    価格比較ロジック
      ↓
    買い物プラン算出ロジック
      ↓
    複数の買い物プランを表示
      ↓
    必要に応じてGoogleマップリンクを表示

第一版では、複雑なルート最適化は行わず、Googleマップのリンクを表示する形式を想定する。

将来的には、Google Maps APIなどを利用し、移動時間や距離を考慮したルート提案機能を追加する。

店舗側の価格登録機能や管理者側の運用機能は、今後の拡張機能として扱う。

- 簡易的な価格データ登録

- ユーザーアカウント登録
- 店舗アカウント登録
- 店舗との実連携
- 自動チラシ取得
- OCR読み取り
- 決済機能
- 本格的なルート最適化
- 高度な分析機能

## 18. 既存サービスとの差別化

既存サービスでは、チラシ情報や特売情報の閲覧が中心となっている場合がある。

本サービスでは、チラシを閲覧するだけでなく、ユーザーが買いたい商品リストを登録し、そのリストに基づいて価格比較や買い物プラン提案を行う点を差別化要素とする。

主な差別化ポイントは以下の通りである。

| 項目         | 既存サービス             | 本サービス                               |
| ------------ | ------------------------ | ---------------------------------------- |
| 主な目的     | チラシ・特売情報の閲覧   | 買い物判断の支援                         |
| 情報の見方   | 店舗ごとのチラシを見る   | 買い物リストをもとに比較する             |
| ユーザー操作 | 自分で見て判断する       | システムが買い物プランを提案する         |
| 比較単位     | 商品単位・店舗単位が中心 | 買い物リスト全体                         |
| 考慮する要素 | 価格・チラシ情報         | 合計金額・店舗数・移動時間               |
| 将来的な拡張 | チラシ閲覧中心           | ルート提案・節約金額表示・店舗側登録支援 |

本サービスは、単なるチラシ閲覧サービスではなく、ユーザーの買い物リストを起点として、より具体的な買い物方法を提案する点で差別化を図る。

## 20. 実装上の課題

本サービスを実装する上で想定される課題は以下の通りである。

| 課題                   | 内容                                                         |
| ---------------------- | ------------------------------------------------------------ |
| データの正確性         | 登録された価格情報が正しいか確認する必要がある               |
| 情報更新               | 特売期間や価格変更に対応する必要がある                       |
| 商品名の表記ゆれ       | 同じ商品でも店舗によって表記が異なる可能性がある             |
| 移動時間の計算         | 地図サービスとの連携が必要になる                             |
| 店舗側の継続利用       | 将来的に店舗側機能を追加する場合、登録作業の負担を減らす必要がある |
| 既存サービスとの差別化 | 単なるチラシ閲覧ではない価値を明確にする必要がある           |

特に、価格情報の取得方法は本サービスの重要な課題である。

第一版では手入力でデータを管理し、将来的に店舗側の登録機能、チラシ画像のOCR読み取り、公開情報の取得などを検討する。

## 24. まとめ

本サービスは、物価上昇による生活費負担の増加を背景に、ユーザーが日常の買い物をより安く、効率的に行えるよう支援することを目的とする。

既存のチラシ・特売情報サービスは、情報を閲覧する点では便利である一方、ユーザーが買いたい商品リストをもとに、合計金額、店舗数、移動時間を考慮した買い物方法を提案する点には改善の余地がある。

本サービスでは、ユーザーが買い物リストを作成し、登録済みの商品価格データをもとに複数の買い物プランを比較できるようにする。

第一版ではユーザー向け機能を中心に実装し、小規模なデータを手入力で管理する。

将来的には、店舗側の価格・特売情報登録機能、Google Maps APIとの連携、OCRによるチラシ画像読み取りなどを追加し、ユーザーと店舗の双方を支援する買い物判断支援サービスを目指す。

本プロジェクトは、Webサービスの企画、要件定義、画面設計、データ設計、実装までを一通り経験するためのポートフォリオとしても活用できる。
