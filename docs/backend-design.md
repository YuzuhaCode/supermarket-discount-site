## 1. バックエンド設計の目的

本ドキュメントでは、近所スーパー買い物支援サービスにおけるバックエンド設計方針を整理する。

本サービスでは、ユーザーが買い物リストを作成し、登録済みの商品価格データをもとに、複数の買い物プランを比較できることを目的とする。

バックエンドでは、商品情報、店舗情報、価格情報を管理し、買い物リストに基づいて最適な買い物プランを算出する役割を担う。

---

## 2. MVPにおける前提条件

MVPでは、実際の店舗連携や外部サービスからの自動データ取得は行わない。

第一版では、サンプルデータを使用し、ユーザーが買い物リストを作成した場合に、価格比較と買い物プラン提案ができる流れを検証する。

### 2.1 MVPでの前提

| 項目         | 内容                            |
| ------------ | ------------------------------- |
| データ       | サンプルデータを使用する        |
| 店舗連携     | 実際の店舗とは連携しない        |
| ユーザー登録 | 実装しない                      |
| データベース | 初期段階では必須ではない        |
| 地図連携     | GoogleマップURLリンクを利用する |
| 認証機能     | 実装しない                      |
| 決済機能     | 実装しない                      |

---

## 3. バックエンドの責務

バックエンドの主な責務は以下の通りである。

| 項目                   | 内容                                                       |
| ---------------------- | ---------------------------------------------------------- |
| 商品情報管理           | 商品名、カテゴリー、規格などを管理する                     |
| 店舗情報管理           | 店舗名、住所、営業時間、位置情報などを管理する             |
| 価格情報管理           | 店舗ごとの商品価格、特売期間、更新日を管理する             |
| 買い物リスト処理       | ユーザーが選択した商品と数量を受け取る                     |
| 価格比較処理           | 複数店舗の商品価格を比較する                               |
| 買い物プラン算出       | 時間優先、最安値優先、バランス重視の買い物プランを算出する |
| Googleマップリンク生成 | 店舗情報をもとに地図リンクを生成する                       |

## 4. データモデル

---

## 4.1 Store / 店舗情報

店舗情報は、スーパーやドラッグストアなどの基本情報を管理する。

| 項目           | 型     | 内容                           |
| -------------- | ------ | ------------------------------ |
| id             | string | 店舗を一意に識別するID         |

| type           | string | 店舗種別                       |
| address        | string | 店舗住所                       |
| latitude       | number | 緯度                           |
| longitude      | number | 経度                           |
| openingHours   | string | 営業時間                       |
| nearestStation | string | 最寄り駅                       |
| walkingTime    | number | ユーザー基準地点からの徒歩時間 |
| mapUrl         | string | GoogleマップURL                |

      "type": "supermarket",

## 6. 買い物プラン算出ロジック

本サービスの中心となる処理は、買い物リストをもとに複数の買い物プランを算出することである。

MVPでは、以下の3種類のプランを算出する。

---

## 6.1 時間優先プラン

### 目的

移動時間を短くし、一店舗でできるだけ買い物を完了できるプランを提示する。

### 算出方針

- できるだけ一店舗で購入する
- ユーザーから近い店舗を優先する
- すべての商品を購入できる店舗を優先する
- 価格よりも移動時間を重視する

### 表示例

| 項目         | 内容                                                   |
| ------------ | ------------------------------------------------------ |
| プラン名     | 時間優先プラン                                         |
| 購入店舗     | スーパーA                                              |
| 合計金額     | 2,300円                                                |
| 移動時間     | 徒歩5分                                                |
| おすすめ理由 | 一店舗で買い物が完了するため、移動時間を短くできます。 |

---

## 6.2 最安値優先プラン

### 目的

商品ごとに最も安い店舗を選び、合計金額を最小化するプランを提示する。

### 算出方針

- 商品ごとに最安値の店舗を選択する
- 移動時間よりも価格を優先する
- 複数店舗を回ることを許容する
- 節約金額を最大化する

### 表示例

| 項目         | 内容                                                         |
| ------------ | ------------------------------------------------------------ |
| プラン名     | 最安値優先プラン                                             |
| 購入店舗     | スーパーA、スーパーB、ドラッグストアC                        |
| 合計金額     | 2,050円                                                      |
| 節約金額     | 250円                                                        |
| 移動時間     | 徒歩30分                                                     |
| おすすめ理由 | 商品ごとに最安値の店舗を選ぶため、合計金額を最も安くできます。 |

---

## 6.3 バランス重視プラン

### 目的

価格と移動時間のバランスを考慮し、現実的に利用しやすいプランを提示する。

### 算出方針

- 合計金額をある程度抑える
- 移動時間が長くなりすぎないようにする
- 店舗数を増やしすぎない
- MVPでは最大2店舗程度を想定する
- 節約金額と移動時間のバランスを評価する

### 表示例

| 項目         | 内容                                                         |
| ------------ | ------------------------------------------------------------ |
| プラン名     | バランス重視プラン                                           |
| 購入店舗     | スーパーA、ドラッグストアC                                   |
| 合計金額     | 2,150円                                                      |
| 節約金額     | 150円                                                        |
| 移動時間     | 徒歩12分                                                     |
| おすすめ理由 | 価格と移動時間のバランスがよく、現実的に利用しやすいプランです。 |

## 7. Googleマップリンク生成

MVPでは、Google Maps APIを利用した本格的なルート最適化は行わない。

第一版では、店舗住所または店舗名をもとに、Googleマップ検索用URLを生成する。

### 生成例

    https://www.google.com/maps/search/?api=1&query=スーパーA

将来的には、Google Maps APIを利用し、複数店舗を回る場合の移動時間や距離を算出することを検討する。

## 12. データ取得方法に関する課題

本サービスを実際に運用する場合、最も重要な課題は価格情報の取得方法である。

価格情報の取得方法としては、主に以下の2つが考えられる。

| 方法                     | 内容                                           | 課題                                               |
| ------------------------ | ---------------------------------------------- | -------------------------------------------------- |
| 外部サイトからの情報取得 | 大型チラシサイトや店舗サイトから情報を取得する | 情報の正確性、利用規約、スクレイピングの問題がある |
| 店舗側登録               | 店舗が自ら価格や特売情報を登録する             | 店舗側の登録負担を減らす必要がある                 |

MVPでは、これらの課題を避けるため、実際の店舗データではなくサンプルデータを使用する。

将来的に本格的なサービスとして展開する場合は、店舗側が簡単に価格や特売情報を登録できる仕組みを検討する必要がある。
