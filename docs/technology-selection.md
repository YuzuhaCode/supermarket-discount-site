## 1. 技術選定の目的

本ドキュメントでは、近所スーパー買い物支援サービスのMVP開発において採用する技術と、その選定理由を整理する。

本サービスでは、ユーザーが商品を検索し、買い物リストを作成し、複数の買い物プランを比較できることを目的とする。

そのため、フロントエンドでは分かりやすいUIを構築し、バックエンドでは商品情報、店舗情報、価格情報を扱い、買い物プランを算出する処理を実装する。

---

## 2. MVPにおける開発方針

MVPでは、実際の店舗連携やデータベース接続よりも、まずはサービス全体の流れを確認できることを優先する。

第一版では、以下を重視する。

- ユーザー向け画面を実装する
- 商品検索、買い物リスト作成、買い物プラン比較の流れを作る
- サンプルデータを使用して動作確認する
- フロントエンドとバックエンドを分離する
- REST APIを通じてデータをやり取りする
- 将来的にデータベースや店舗側機能を追加できる構成にする

## 4. フロントエンド技術

## 4.1 React

フロントエンドにはReactを採用する。

Reactを採用する理由は以下の通りである。

- コンポーネント単位でUIを分割できる
- 商品テーブル、買い物リスト、プランカードなどを再利用しやすい
- 状態管理を使って買い物リストの追加、数量変更、削除を実装しやすい
- バックエンドAPIとの連携を追加しやすい
- 実務でも利用されることが多く、ポートフォリオとして説明しやすい

本サービスでは、以下のようなコンポーネントを想定する。

- Header
- Button
- ProductTable
- ProductSearchFilter
- ShoppingListTable
- SummaryCard
- PlanCard
- StoreInfoCard

---

## 4.2 Parcel

Reactの開発環境にはParcelを採用する。

Parcelを採用する理由は以下の通りである。

- 設定が比較的シンプルである
- Reactアプリを素早く構築できる
- MVP開発に必要な構成を最小限で用意できる
- フロントエンドとJava / Spring Bootバックエンドを分離して開発しやすい
- 学習コストを抑えながら、Reactの画面実装に集中できる

本プロジェクトでは、React + Parcelでフロントエンドを構築し、Java / Spring Bootで作成したREST APIと連携する。

---

## 4.3 スタイリング

スタイリングには、CSS Modulesまたは通常CSSを使用する。

MVPでは、過度に複雑なCSS設計よりも、画面を分かりやすく実装することを優先する。

スタイリング方針は以下の通りである。

- 白背景を基調とする
- グリーン系をメインカラーにする
- 価格や節約金額は赤系またはオレンジ系で強調する
- カード、テーブル、ボタンのデザインを統一する
- PC向けWebサイトとして見やすい余白を確保する

---

## 5. バックエンド技術

## 5.1 Java / Spring Boot

バックエンドにはJava / Spring Bootを採用する。

Spring Bootを採用する理由は以下の通りである。

- REST APIを作成しやすい
- Controller、Service、Modelなどの役割を分けて実装しやすい
- 商品情報、店舗情報、価格情報などのデータ構造を整理しやすい
- 買い物プラン算出ロジックをService層にまとめやすい
- 将来的にデータベースや認証機能を追加しやすい
- Javaは企業システムで利用されることが多く、就職活動用のポートフォリオとして説明しやすい

本サービスでは、以下のような構成を想定する。

- Controller：APIの入口
- Service：買い物プラン算出などの業務ロジック
- Model：商品、店舗、価格、買い物リストなどのデータ構造
- Repository：将来的なデータ取得処理

---

## 5.2 REST API

フロントエンドとバックエンドの通信にはREST APIを採用する。

REST APIを採用する理由は以下の通りである。

- ReactとJavaバックエンドを分離しやすい
- JSON形式でデータをやり取りできる
- API単位で機能を確認しやすい
- 将来的にスマートフォンアプリや店舗側画面を追加する場合にも再利用しやすい

MVPで想定するAPIは以下の通りである。

| メソッド | エンドポイント             | 内容                                       |
| -------- | -------------------------- | ------------------------------------------ |
| GET      | /api/products              | 商品一覧を取得する                         |
| GET      | /api/stores                | 店舗一覧を取得する                         |
| GET      | /api/prices                | 価格一覧を取得する                         |
| POST     | /api/shopping-list/compare | 買い物リストをもとに買い物プランを算出する |

---

## 6. データ管理方針

MVPでは、データベースは使用せず、サンプルデータを使用する。

理由は以下の通りである。

- 最初からデータベースを導入すると実装範囲が広がるため
- まずは画面とAPIの流れを確認したいため
- 商品、店舗、価格データの構造を先に検証したいため
- 買い物プラン算出ロジックを優先して実装したいため

MVPで扱う主なデータは以下の通りである。

| データ       | 内容                             |
| ------------ | -------------------------------- |
| Product      | 商品名、カテゴリー、規格         |
| Store        | 店舗名、住所、営業時間、徒歩時間 |
| Price        | 商品ごとの店舗別価格             |
| ShoppingList | ユーザーが選択した商品と数量     |
| ShoppingPlan | 算出された買い物プラン           |

将来的には、PostgreSQLを導入し、店舗、商品、価格情報を永続化する。

---

## 7. 買い物プラン算出ロジック

本サービスのバックエンドで最も重要な処理は、買い物リストをもとに買い物プランを算出することである。

MVPでは、以下の3種類のプランを算出する。

| プラン             | 内容                                                         |
| ------------------ | ------------------------------------------------------------ |
| 時間優先プラン     | できるだけ近い店舗、または一店舗で買い物を完了できるプラン   |
| 最安値優先プラン   | 商品ごとに最も安い店舗を選び、合計金額を最小化するプラン     |
| バランス重視プラン | 価格と移動時間のバランスを考慮し、現実的に利用しやすいプラン |

このロジックは、Spring BootのService層に実装する。

---

## 8. 地図連携方針

MVPでは、Google Maps APIを利用した本格的なルート計算は行わない。

第一版では、店舗名または住所をもとにGoogleマップの検索URLを生成する。

例：

    https://www.google.com/maps/search/?api=1&query=スーパーA

将来的には、Google Maps APIを利用し、複数店舗を回る場合の移動時間やルート最適化を検討する。

## 10. 採用しない技術と理由

MVPでは、以下の技術や機能は採用しない。

| 技術・機能       | 採用しない理由                                               |
| ---------------- | ------------------------------------------------------------ |
| Next.js          | 今回はReactとJavaバックエンドを分離して実装するため。        |
| Vite             | 今回はよりシンプルな構成でReactアプリを構築するため、Parcelを採用する。 |
| Create React App | 新規プロジェクトでは採用せず、より軽量な構成を選択するため。 |
| データベース     | MVPではサンプルデータで画面とAPIの流れを確認するため。       |
| 認証機能         | 第一版ではユーザー登録やログインを必要としないため。         |
| 店舗側管理画面   | MVPではユーザー向け機能を優先するため。                      |
| Google Maps API  | 第一版ではGoogleマップURLで十分なため。                      |
| スクレイピング   | 利用規約、著作権、データ正確性の課題があるため。             |
| OCR              | チラシ画像読み取りは実装難易度が高く、MVP範囲外のため。      |
| 決済機能         | 本サービスは購入判断支援サービスであり、決済機能は不要なため。 |

## 13. まとめ

本プロジェクトでは、フロントエンドにReact、フロントエンド開発環境にParcel、バックエンドにJava / Spring Bootを採用する。

Reactでは、商品検索、買い物リスト、プラン比較などの画面をコンポーネント単位で実装する。

Parcelを使用することで、設定を複雑にしすぎず、Reactアプリの開発環境を用意する。

Spring Bootでは、商品情報、店舗情報、価格情報を扱うREST APIを作成し、買い物リストをもとに3種類の買い物プランを算出する。

MVPでは、データベースや店舗連携は行わず、サンプルデータを使用して、サービス全体の流れと中核となる買い物プラン算出ロジックを検証する。

将来的には、PostgreSQL、店舗側登録機能、Google Maps API連携などを段階的に追加し、より実用的なサービスへ拡張する。
